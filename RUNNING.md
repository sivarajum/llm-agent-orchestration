# LLM Agent Orchestration - Running Guide

A multi-agent system built on LangGraph that orchestrates three AI agents (Researcher, Writer, Reviewer) in a state-machine loop to produce researched articles on any topic.

## Prerequisites

- Python 3.11+
- pip
- Docker and Docker Compose (optional, for containerized deployment)

## Install

```bash
cd Sj-Prod/LLM-Agent-Orchestration
pip install -r requirements.txt
```

## Environment Setup

Copy the example env file and (optionally) add your API keys:

```bash
cp .env.example .env
```

Edit `.env` to add one of:

```
OPENAI_API_KEY=sk-...
# OR
ANTHROPIC_API_KEY=sk-ant-...
```

**API keys are optional.** The system works fully without them using built-in fallback/offline mode. With keys, agents use GPT-4o-mini or Claude Sonnet for higher-quality output.

## How to Run a Topic (CLI)

Run the full Researcher -> Writer -> Reviewer pipeline directly from the terminal:

```bash
python main.py run "Quantum Computing"
python main.py run "AI agents in production"
python main.py run "Data engineering best practices"
```

This prints the research, draft, and review feedback to stdout.

## How to Run the System

The system supports four modes via `main.py`:

```bash
# Start the FastAPI server on port 8000 (default)
python main.py api

# Start the Streamlit dashboard on port 8501
python main.py ui

# Start both API and UI together
python main.py all

# Run pipeline directly in the terminal
python main.py run "your topic here"
```

The UI connects to the API, so when running `ui` standalone, the API must already be running.

## How to Run with Docker

```bash
# Build and start both API and UI
docker compose up --build

# Start in detached mode
docker compose up --build -d

# Stop
docker compose down
```

Services:
- API: http://localhost:8000
- UI: http://localhost:8501

To pass API keys via Docker:

```bash
OPENAI_API_KEY=sk-... docker compose up --build
```

Or create a `.env` file in the project root (Docker Compose reads it automatically).

## Fallback Mode

When no LLM API keys are set, each agent uses deterministic fallback logic:

- **Researcher**: Attempts DuckDuckGo web search. If search also fails (no internet), returns a canned research brief with general observations about the topic.
- **Writer**: Generates a template-based markdown article with Introduction, Key Findings, Analysis, and Conclusion sections extracted from the research.
- **Reviewer**: Uses heuristic scoring based on word count, heading count, conclusion presence, and paragraph structure. Scores range from 5-10.

This makes the system fully functional for development, testing, and demos without any external API dependencies.

## Architecture

The system implements a LangGraph state machine with three agent nodes:

```
START -> Researcher -> Writer -> Reviewer -> END
                         ^                    |
                         |   (if score < 7    |
                         |    and iter < 3)   |
                         +--------------------+
```

### State Machine (AgentState)

A TypedDict flowing through the graph with fields:
- `messages` - Chat message history (LangGraph message accumulator)
- `topic` - User-provided topic string
- `research` - Output from Researcher agent
- `draft` - Output from Writer agent
- `review_feedback` - Output from Reviewer agent
- `iteration` - Current revision count (0-3)
- `status` - Pipeline stage: researching | writing | reviewing | complete

### Agent Nodes

| Agent | File | Role |
|-------|------|------|
| Researcher | `src/agents/researcher.py` | Searches the web, summarizes findings |
| Writer | `src/agents/writer.py` | Produces a structured markdown article |
| Reviewer | `src/agents/reviewer.py` | Scores the draft (1-10), approves or requests revision |

### Conditional Edge

After each review, the `_should_revise` function routes:
- **Back to Writer** if `status != "complete"` (score < 7 and iteration < 3)
- **To END** if approved (score >= 7) or max iterations reached (iteration >= 3)

### Supporting Modules

| Module | Purpose |
|--------|---------|
| `src/state.py` | AgentState TypedDict definition |
| `src/orchestrator.py` | Graph construction and `run_pipeline()` entry point |
| `src/llm.py` | LLM factory (OpenAI/Anthropic with fallback) |
| `src/tools.py` | Web search and calculator tools |
| `src/api.py` | FastAPI server |
| `src/ui.py` | Streamlit dashboard |
| `main.py` | CLI entry point |

## Production Deployment Notes

### Scaling

- The API server is stateless and can be horizontally scaled behind a load balancer.
- Each pipeline invocation is independent; no shared state between requests.
- For high throughput, consider adding a task queue (Celery/Redis) for async pipeline execution.

### Timeouts

- The pipeline can take 15-60 seconds depending on LLM response times and iteration count.
- The Streamlit UI uses a 120-second httpx timeout.
- Configure reverse proxy timeouts accordingly.

### Security

- CORS is currently set to allow all origins (`*`). Restrict this in production.
- API keys should be provided via environment variables, never hardcoded.
- The `calculate` tool uses a restricted `eval()` -- only numeric characters and operators are allowed.

### Monitoring

- Use the `/health` endpoint for liveness/readiness probes.
- Docker Compose includes a healthcheck for the API service.
- Add structured logging and metrics (e.g., OpenTelemetry) for production observability.

### Configuration

- LLM model selection is in `src/llm.py` (currently gpt-4o-mini / claude-sonnet-4-20250514).
- Max review iterations (3) is hardcoded in `src/agents/reviewer.py`.
- Approval threshold (score >= 7) is in `src/agents/reviewer.py`.
